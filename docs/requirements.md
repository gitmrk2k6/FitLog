# 要件定義書 — FitLog（筋トレ・運動記録アプリ）

## 1. 概要・目的

### 1.1 背景

RaiseTech AIエンジニアコース **中級編の最終課題**。中級編の学習教材 RaiseTimeLine（X風SNS）を踏まえ、自作アプリを開発する。
課題要件は「**複数ユーザーが双方向にデータをやり取りするサービス**」であること（SNSである必要はない）。
技術スタックは教材（RaiseTimeLine）と変更する（AWSは教材と同じ構成でよい）。

### 1.2 目的

- 筋トレ/運動の記録（種目・セット・重量・回数・写真）を蓄積し、週間/月間で集計・可視化する
- 記録する人が **継続できる工夫**（目標達成率・ストリーク・自己ベスト）でモチベーションを維持する
- 複数ユーザーが互いの記録を見て応援し合う双方向のやり取り（ナイストレ/アドバイス/フォロー）を提供する

### 1.3 本書の位置づけ

本ドキュメントは **要件定義の目次（ハブ）**。詳細は各ドキュメントに分割して管理する。

---

## 2. 想定ユーザー

| ペルソナ | 概要 | 主な利用シーン |
| --- | --- | --- |
| 継続したい個人 | 筋トレ/運動を習慣化したいが続かない人 | 毎回の記録、進捗の可視化でモチベ維持 |
| 受講生/学習者 | RaiseTech を受講している学習者 | 学習の合間の運動記録、仲間との励まし合い |

※ 複数ユーザーが同時に利用し、互いの記録に対して双方向にやり取りすることを前提に設計する。

## 3. 機能要件（概要）

詳細な機能定義・バリデーション・ユースケースは [機能要件書](functional-requirements.md) を参照。

| 区分 | 機能ID | 機能名 | 概要 |
| --- | --- | --- | --- |
| MVP | F-01 | ユーザー認証 | 登録・ログイン・ログアウト・JWT認証 |
| MVP | F-02 | トレーニング記録 | 実施日・種目・セット（重量×回数）・メモ・写真の記録/編集/削除 |
| MVP | F-03 | 記録一覧/詳細 | 自分の記録を日別に時系列表示、詳細閲覧、総ボリューム併記 |
| MVP | F-04 | ナイストレーニング | 他ユーザーの記録への「ナイストレ」（いいね相当）・解除・件数表示 |
| MVP | F-05 | アドバイス・応援 | 他ユーザーの記録へのコメント（応援/助言）・削除 |
| MVP | F-06 | トレーニング仲間フォロー | ユーザー検索・フォロー/アンフォロー・フォロー中フィード |
| MVP | F-07 | 目標設定＋達成率 | 週間/月間目標（回数 or 総ボリューム）の設定と達成率グラフ |
| MVP | F-08 | ストリーク＋ヒートマップ | 連続記録日数とカレンダーヒートマップで習慣を可視化 |
| MVP | F-09 | 自己ベスト(PR)自動検出 | 記録時に種目別の過去最大重量/推定1RM超過を判定し「自己ベスト更新」表示 |
| MVP | F-10 | 写真の保存 | 進捗/フォーム写真を添付し AWS S3 に保存（教材と同じ構成を流用） |

## 4. 非機能要件

| カテゴリ | 要件 |
| --- | --- |
| 性能 | 通常操作（表示・記録・集計）が2秒以内に応答。集計クエリはインデックス（user_id, performed_on）前提 |
| セキュリティ | パスワードはハッシュ化（Bcrypt）。JWT認証により自分のデータのみ操作可能 |
| 可用性 | 学習用途のため高可用構成は対象外（単一構成で動作すれば可） |
| 保守性 | バックエンドは router / service / repository の層構造。機能実装はテストとセットで完了とする |
| 互換性 | モダンブラウザ（Chrome, Firefox, Edge の最新版）で動作 |
| ストレージ | 写真は AWS S3 に保存し、DBにはURLのみ保持。アプリサーバーには保持しない |

---

## 5. 技術スタック（概要）

FastAPI(Python) + Vue 3 + PostgreSQL を軸に構成。教材（Java/Spring + Next.js/React + MyBatis）からの変更点・選定理由は
[技術スタック](tech-stack.md) を参照。AWS は教材と同じ構成（ECS Fargate + CloudFront + S3 + RDS + ALB）を流用する。

---

## 6. データベース設計（概要）

`users` を中心に、`workouts`（実施日単位の記録）→ `workout_sets`（セット）→ `exercises`（種目マスタ）の階層。
`cheers`/`advices` が記録への双方向アクション、`follows` がユーザー間関係、`goals` が目標を管理。
ER図・テーブル定義・インデックス・制約は [データベース設計書](database-design.md) を参照。

---

## 7. 画面設計（概要）

ログイン・新規登録・ダッシュボード（達成率/ストリーク/ヒートマップ/PR）・記録入力・記録一覧/詳細・集計・目標設定・
フィード/ユーザー検索 を主軸とする。画面一覧・遷移図・ワイヤーフレームは [画面設計書](screen-design.md) に作成済み。

---

## 8. インフラ構成（概要）

教材 RaiseTimeLine と同じ AWS 構成（ECS Fargate + CloudFront + S3 + RDS + ALB）を流用。
詳細は後フェーズで [インフラ構成](infrastructure.md) に作成する。

---

## 9. スコープ外（今回は実施しない）

- 通知機能（ナイストレ・アドバイス・フォローのメール/Push通知）
- 実績バッジの多数展開（PR検出以外のゲーミフィケーション）
- モバイルネイティブアプリ対応
- 外部サービス連携（Googleログイン等）
- DM（ダイレクトメッセージ）機能
